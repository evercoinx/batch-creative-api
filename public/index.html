Batch Creative API

One on-brand social post per product image — generated as a reliable, async batch with provider retries and failover.

1 · The product

Interactive demo — running in mock mode. Pick product images and a reference style, choose a platform, and submit a batch.

2 · How it was built

Approach & key decisions

The service is a single async HTTP API behind a vanilla demo page. POST /batches validates the request with zod, creates an in-memory batch, and returns a batch id immediately; the browser polls GET /batches/:id and watches per-item status go from pending → running → done/failed. All image generation flows through one ImageProvider seam, with a resilience layer that retries transient errors (3× exponential backoff + jitter), times out each call, and fails over from the primary provider to the fallback so a single outage never sinks the batch. One style spec is extracted per batch from the reference images and reused for every product, keeping the set visually consistent. Built and grilled decision-by-decision rather than from a prototype; the architecture is recorded in docs/adr/0001.

How AI was used

Built with Claude (Opus 4.8) driving a test-first RGR loop: each slice started with a failing test, then the implementation to make it pass, then a refactor pass. AI also drafted the domain glossary, the architecture ADR, and these write-ups, and acts as the runtime — Gemini (Imagen + gemini-2.5-flash) is the primary image/caption provider with OpenAI (gpt-image-1 + gpt-4o-mini) as the failover.

Toolset

  • Runtime: Node 22 running .mts via --experimental-strip-types
  • Web: Express 5 + zod for request validation
  • Providers: @google/genai (Gemini), openai SDKs behind one seam
  • Tests: vitest — retry/failover/lifecycle covered with a scripted mock provider, no network
  • Deploy: Render web service via committed render.yaml; ephemeral disk for outputs/
